# GameFi 经济系统设计文档 V0.2

版本：V0.2  
日期：2026-07-10  
状态：内部设计稿，可继续拆分为白皮书、后端需求、运营参数表  

## 1. 文档目标

本文档用于定义当前 GameFi 项目的经济系统，包括 Token 发行、钱包分配、游戏内三层货币、产出与消耗、交易行、NFT、提现冷却、贡献等级、Revenue 回流、热钱包发奖、风控和测试期参数。

这不是宣传稿，而是内部执行文档。它的目标是让开发、运营、推手、社区和项目方都能理解：

- 项目方怎么赚钱。
- 玩家为什么愿意玩。
- GAME Token 为什么有需求。
- 游戏产出如何限制。
- 初始 LP 很小的情况下如何避免被快速抽干。
- 哪些参数必须可后台调整。
- 哪些资产必须放在冷钱包、多签或人工管理钱包。
- 哪些资产可以放在后端热钱包。

## 2. 项目经济总原则

### 2.1 不做无限 mint

GAME 总量固定为 10 亿枚，一次性铸造，随后关闭 mint 权限。未来服务器增加、玩家增加、活动增加，都不通过新增 mint 解决，而是通过预留钱包释放。

这样做的好处：

- 对外信任更强。
- 玩家不用担心项目方后续无限增发。
- 经济模型更容易解释。
- 合约更简单，安全风险更低。

### 2.2 游戏经济优先，币价叙事其次

GAME 的核心价值应该来自游戏内用途，而不是只靠二级市场炒作。

GAME 必须有以下需求来源：

- 成长系统消耗。
- 战斗系统消耗。
- 交易行买卖。
- NFT Mint。
- 提现权益和贡献等级。
- Revenue 回流带来的回购和 LP 增深。

### 2.3 前期小口子测试

当前现金预算约 2kU，开发和人力成本可以减免，但运营、推手、测试玩家、流动性都需要谨慎使用资金。

因此前期不能开放大产出、大提现、大空投。正确方式是：

- 小规模真实测试。
- 小额度产出。
- 小额度提现。
- 冷却机制。
- 小额自动，大额人工。
- 数据驱动调参。

### 2.4 收入和 Token 价值分开设计

项目方收入主要来自：

- 宝石充值。
- NFT Mint。
- NFT 版税。
- 游戏内付费内容。
- 合作收入。
- 活动赞助。

Token 价值主要来自：

- 游戏内消耗。
- 交易行需求。
- NFT Mint 需求。
- Revenue 回购。
- LP 深度增长。
- 产出冷却和提现限额控制卖压。

不要把项目收入设计成单纯卖 GAME。卖 GAME 只是流动性启动，不应该是长期收入模型。

## 3. 资产体系

项目内资产分为三层货币和若干物品资产。

### 3.1 GAME Token

GAME 是链上 Token，是项目经济核心资产。

用途：

- 游戏内系统消费。
- 成长系统升级。
- 战斗系统门票和强化。
- 游戏交易行唯一交易货币。
- 稀有物 NFT Mint。
- 贡献等级计算。
- 提现和外部交易。
- Revenue 回购和 LP 配对资产。

GAME 分类：

- 外部充值 GAME：玩家从市场买入后充值进游戏，不进入冷却。
- 产出 GAME：玩家从游戏系统、交易行、活动、Boss、Gold 兑换中获得，进入冷却。
- 可提现 GAME：已过冷却，可提到链上。
- 冷却中 GAME：可在游戏内消费，但不能提现。

### 3.2 宝石

宝石是等价 U 的充值点数，不是链上 Token。

规则：

- 只进不出。
- 不能提现。
- 不可转账。
- 不可兑换 GAME。
- 不可兑换 USDC / SOL / U。
- 一般按商品价格精准充值并立刻消费。
- 只有活动奖励、双倍奖励、补偿、返利时可能出现余额。

用途：

- 月卡。
- 通行证。
- 皮肤。
- 外观。
- 礼包。
- 便利功能。
- 改名。
- 活动商品。
- 特殊门票。

宝石收入进入 Revenue Wallet，再按 Revenue 规则处理。

### 3.3 Gold / 金币

Gold 是游戏内软货币，不上链。

来源：

- 日常采集。
- 副本。
- Boss。
- 任务。
- 活动。

用途：

- 基础成长。
- 合成。
- 升级。
- 兑换 GAME。

当前兑换比例：

```text
10 Gold = 1 GAME
```

Gold 转 GAME 没有额外冷却。冷却发生在 GAME 领取后，到可提现之间。

### 3.4 物品资产

物品分为：

- 普通材料。
- 稀有材料。
- 顶级稀有物。
- 装备。
- NFT。

规则：

- 普通材料不交易。
- 稀有材料可以进入游戏交易行。
- 顶级稀有物可以进入游戏交易行，也可以 Mint NFT。
- NFT 可以进入外部市场。

## 4. GAME Token 发行与分配

### 4.1 总量

```text
总量：1,000,000,000 GAME
```

### 4.2 权限

Token 发行后建议关闭：

- mint 权限。
- 不必要的管理员增发权限。
- 黑名单、冻结、税率修改等高信任风险权限，如无必要不保留。

如果确实要保留某些权限，必须公开说明用途和限制，并优先使用多签。

### 4.3 分配表

| 分类 | 数量 | 比例 | 用途 |
|---|---:|---:|---|
| Initial LP | 20,000,000 | 2% | 初始流动性 |
| Team Locked Wallet | 250,000,000 | 25% | 团队锁仓 |
| Treasury Wallet | 320,000,000 | 32% | 长期储备、运营安全垫 |
| Game Rewards Wallet | 250,000,000 | 25% | 游戏奖励、测试玩家、赛季发放 |
| Growth Wallet | 160,000,000 | 16% | KOL、推手、合作、后期流动性、服务器增长 |
| Revenue Wallet | 0 | 0% | 接收外部收入，不预分 GAME |

### 4.4 团队锁仓

团队分配：

```text
250,000,000 GAME
25%
```

释放规则：

- 锁定 6 个月。
- 之后 24 个月线性释放。
- 前 6 个月团队不卖出。
- 团队锁仓地址公开。

目的：

- 提升信任。
- 降低早期砸盘担忧。
- 保留长期激励。

## 5. 初始 LP 设计

### 5.1 初始池子

```text
2,000 USDC + 20,000,000 GAME
```

初始价格：

```text
1 GAME = 0.0001 USDC
```

初始 FDV：

```text
1,000,000,000 * 0.0001 = 100,000 USDC
```

LP Token 侧占总量：

```text
20,000,000 / 1,000,000,000 = 2%
```

### 5.2 初始 LP 风险

2,000U 初始流动性很浅。

影响：

- 少量买单会明显拉涨。
- 少量卖单会明显砸跌。
- 产出提现必须非常保守。
- 交易行收益和游戏产出必须进入冷却。
- 热钱包不能放太多 GAME。
- 全服每小时提现和每日提现必须有限额。

### 5.3 后续增深 LP

后续通过 Revenue Wallet 的增长回流来增深 LP。

默认：

```text
Revenue 的 40% 用于增长回流
其中 20% 回购 GAME
其中 20% 加 LP
```

前期人工执行，必须记账。

不建议频繁主观调价格。更合理的方式是：

- 收入增长后加深 LP。
- 价格弱时提高回购比例。
- 价格过热且池子浅时提高 LP 增深比例。
- 所有操作留记录。

## 6. 钱包体系

### 6.1 必备钱包

前期至少需要以下钱包：

| 钱包 | 类型 | 用途 | 是否热钱包 |
|---|---|---|---|
| Team Locked Wallet | 锁仓钱包 | 团队 Token | 否 |
| Treasury Wallet | 金库钱包 | 长期储备 | 否 |
| Game Rewards Wallet | 奖励储备钱包 | 游戏奖励储备 | 否 |
| Growth Wallet | 增长钱包 | KOL、推手、合作、测试激励 | 否 |
| Revenue Wallet | 收入钱包 | 接收外部收入 | 否 |
| Reward Distribution Wallet | 奖励分发热钱包 | 后端自动发奖和小额提现 | 是 |

### 6.2 Treasury Wallet

Treasury 是长期金库。

用途：

- 长期运营储备。
- 极端市场情况下支持经济。
- 人工转入热钱包预算。
- 后续迁移多签。
- 长期资金管理。

规则：

- 不暴露私钥给后端。
- 不直接自动发奖。
- 不做频繁小额支付。
- 后期迁移多签。

### 6.3 Revenue Wallet

Revenue 是收入入口。

收入来源：

- 宝石充值。
- NFT 版税。
- 合作收入。
- 活动赞助。
- 其他 USDC / SOL 收入。

Revenue 分配：

```text
30% 运营
40% 增长回流
20% Treasury
10% 玩家活动
```

增长回流默认：

```text
20% 回购 GAME
20% 加 LP
```

### 6.4 Reward Distribution Wallet

Reward Distribution Wallet 是后端热钱包，只放少量预算。

规则：

- 平时放 2-3 天自动奖励预算。
- 余额低于 0.5 天预算时报警。
- 活动单独充值预算。
- 活动结束后剩余转回。
- 热钱包耗尽时自动发奖暂停。

必须限制：

- 单笔发放上限。
- 单用户每日发放上限。
- 全服每小时提现上限。
- 全服每日提现上限。
- 后端订单号防重复发放。
- 异常自动暂停。

## 7. Revenue 流转

### 7.1 收入路径

```text
玩家充值宝石 / NFT版税 / 合作收入
        |
        v
Revenue Wallet
        |
        +--> 30% 运营
        +--> 40% 增长回流
        |        +--> 默认 20% 回购 GAME
        |        +--> 默认 20% 加 LP
        |
        +--> 20% Treasury
        +--> 10% 玩家活动
```

### 7.2 执行原则

- 前期人工执行。
- 每次分配记录金额、时间、用途、tx hash。
- 不拆太多收入钱包，避免混乱。
- Revenue 先统一收，再按规则分配。
- 特殊情况允许调整，但必须说明原因。

## 8. 服务器与全局账本

### 8.1 服务器定义

服务器不是独立经济体，而是：

- 玩法实例。
- 社交分流。
- 性能分流。
- 压力分摊。

玩家数据全服互通。

### 8.2 全局经济账本

以下必须全局统一：

- GAME 余额。
- Gold 余额。
- 宝石余额。
- Gold 兑换额度。
- 冷却记录。
- 提现额度。
- 交易行记录。
- 奖励池。
- 风控额度。

不能按服独立经济，否则会出现跨服套利。

### 8.3 服务器扩容

服务器人工增加。

新增条件：

- 现有服务器连续一周平均活跃达到 70%。
- 单服上限暂定 50 人。
- 达到条件后考虑新增。

减少服务器：

- 活跃不足可以减少服务器。
- 减少服务器不 burn Token。
- 未来恢复到历史服务器数量时，不再次释放增长预留。
- 只有超过历史最高服务器数量，才可释放新的增长预算。

## 10. 资源与稀有掉落模型

采用：

```text
基础保底 + 高级竞争 + Boss 贡献
```

### 10.1 基础资源

基础资源提供个人保底。

目的：

- 晚上线玩家仍有资源。
- 休闲玩家不会完全陪跑。
- 游戏前期体验稳定。

### 10.2 高级资源

高级资源可以竞争。

目的：

- 制造社交。
- 制造冲突。
- 制造稀缺。
- 给高活跃玩家目标。

### 10.3 稀有掉落

规则：

- 所有玩家都有概率。
- 概率都很低。
- 不因上线时间改变概率。
- 肝度通过有效次数和碎片保底体现。
- 直接掉落和碎片保底并存。
- 每日/每周有控量。

建议：

```text
每次有效行为 -> 独立低概率判定 -> 未掉落给绑定碎片 -> 碎片达到数量可合成
```

## 11. Gold -> GAME

### 11.1 基础比例

```text
10 Gold = 1 GAME
```

### 11.2 转换限制

需要限制：

- 单玩家每日兑换上限。
- 全服每日兑换上限。
- 玩家 Gold 存储上限。
- 转换效率上限。
- 转换效率升级成本。

初期建议：

- 单玩家每日兑换：100-300 GAME。
- 全服每日兑换：10,000-20,000 GAME。

### 11.3 存储上限

玩家最多可存 5 天可兑换 Gold。

可通过以下方式提升：

- 等级。
- 系统升级。
- GAME 消耗。
- 材料消耗。
- 特定任务。

### 11.4 转换效率

转换效率可升级。

升级需要：

- 玩家等级。
- Gold。
- 材料。
- GAME。
- 可能需要宝石或通行证权益。

在线有效活跃可以增加一定效率，但必须有上限。

## 12. GAME 消耗

### 12.1 成长系统

GAME 可用于：

- 提升 Gold 转 GAME 效率。
- 提升 Gold 存储上限。
- 扩背包。
- 扩仓库。
- 提升采集效率。
- 提升在线效率。
- 解锁高级区域。

### 12.2 战斗系统

GAME 可用于：

- 副本门票。
- Boss 次数。
- 高级资源区门票。
- 装备强化。
- 装备修理。
- 装备洗练。
- 装备合成。
- 失败保护。

### 12.3 消费后分配

前期规则：

```text
10% burn
80% 回收
10% 奖励
```

说明：

- burn 用于轻微通缩。
- 回收进入受控回收池。
- 奖励进入奖励池。

回收池不能直接无限返还玩家，应由项目方按赛季、活动、Boss、排行榜等方式逐步释放。

后期可调整：

- 卖压大：提高 burn。
- 产出不足：提高奖励。
- 活跃低：增加活动释放。
- Token 过度稀缺：降低 burn。

## 13. 交易行

### 13.1 可交易资产

普通材料：

- 不交易。

稀有材料：

- 可进入游戏交易行。

顶级稀有物：

- 可进入游戏交易行。
- 可 Mint NFT。

### 13.2 交易货币

交易行只使用 GAME。

不使用：

- Gold。
- 宝石。
- USDC。
- SOL。

### 13.3 手续费

交易行手续费：

```text
5% GAME
```

分配：

```text
1% burn
3% Treasury / 回收池
1% Rewards
```

后续可调：

- 市场冷：2-3%。
- 正常：5%。
- 倒卖严重：6-8%。
- 卖压大：提高 burn 占比。

### 13.4 交易行收入冷却

交易行卖出获得的 GAME 视为产出收益。

规则：

- 进入 locked GAME。
- 按贡献等级冷却。
- 冷却中可游戏内消费。
- 冷却中消费不计贡献值，不产生即时解锁额度。

## 14. NFT

### 14.1 Mint 条件

顶级稀有物可以消耗 GAME Mint 成 NFT。

建议 Mint 费用：

| 稀有度 | Mint 费用 |
|---|---:|
| 稀有 | 500 - 2,000 GAME |
| 史诗 | 2,000 - 10,000 GAME |
| 传说 | 10,000+ GAME |

### 14.2 外部交易

NFT 可进入外部市场交易。

版税：

- 建议 3-5%。
- 若市场支持则设置。
- 若不强制版税，不把它当核心收入。

版税收入：

```text
外部市场 -> Revenue Wallet -> Revenue 统一分配
```

## 15. 提现冷却与贡献等级

### 15.1 冷却定义

冷却指：

```text
玩家获得 GAME 后，从游戏内到账到可提现/可转出的时间差
```

进入冷却的资产：

- Gold 兑换 GAME。
- 副本奖励。
- Boss 奖励。
- 活动奖励。
- 交易行卖出收益。
- 测试奖励。
- 系统奖励。

不进入冷却：

- 外部市场买入后充值进游戏的 GAME。

### 15.2 余额结构

玩家余额应拆为：

```text
可游戏内消费 GAME
可提现 GAME
冷却中 GAME
```

冷却中 GAME 展示：

- 来源。
- 数量。
- 创建时间。
- 解锁时间。
- 当前状态。

### 15.3 贡献等级

贡献等级永久，但权益按近期活跃维持。

前期冷却范围：

```text
最高 74 小时
最低 24 小时
```

等级表：

| 等级 | 累计有效系统消费 | 提现冷却 |
|---|---:|---:|
| Tier 0 | 0 GAME | 74 小时 |
| Tier 1 | 10,000 GAME | 68 小时 |
| Tier 2 | 50,000 GAME | 60 小时 |
| Tier 3 | 200,000 GAME | 48 小时 |
| Tier 4 | 500,000 GAME | 36 小时 |
| Tier 5 | 1,000,000 GAME | 24 小时 |

### 15.4 权益维持

规则：

- 等级永久不降。
- 最近 30 天无有效活跃或有效系统消费，冷却权益临时回落 1 档。
- 恢复活跃后恢复权益。

### 15.5 即时解锁额度

有效系统消费的 10% 转化为即时解锁额度。

例如：

```text
有效系统消费 10,000 GAME
获得 1,000 GAME 即时解锁额度
```

该额度只能解锁自己的 locked GAME。

不能：

- 转让。
- 交易。
- 提现为独立资产。

### 15.6 有效系统消费

计入贡献等级和即时解锁额度：

- 外部充值 GAME。
- 已解锁 GAME。

不计入：

- 冷却中 GAME。
- 交易行买卖。
- 玩家间转账。
- 系统赠送。
- 活动奖励。
- 补偿。
- pending / locked 产出。

核心防循环规则：

```text
只有外部充值 GAME + 已解锁 GAME 的系统消费，才产生贡献值和解锁额度。
```

## 16. 解锁技术实现

### 16.1 推荐模式

采用：

```text
记录式锁仓 + 定时任务 + 用户触发二次结算
```

不要每小时遍历所有玩家改余额。

### 16.2 锁仓记录表

每笔产出 GAME 生成一条记录：

```text
id
player_id
amount
source
created_at
unlock_at
status
order_ref
tx_hash
```

source：

- gold_convert
- boss
- dungeon
- marketplace
- event
- tester_reward
- manual_reward
- compensation

status：

- locked
- unlocked
- consumed
- withdrawn
- cancelled

### 16.3 定时任务

任务逻辑：

- 前期每小时扫一次。
- 后续可改为每 5-10 分钟。
- 查询 `unlock_at <= now AND status = locked`。
- 将到期记录改为 unlocked。
- 增加玩家 withdrawable_balance。

数据库索引：

- status。
- unlock_at。
- player_id。

### 16.4 用户触发结算

当玩家：

- 打开钱包页。
- 发起提现。
- 查看冷却详情。

系统应即时结算该玩家已到期记录，避免定时任务延迟影响体验。

### 16.5 冷却中 GAME 消费

允许冷却中 GAME 在游戏内消费。

规则：

- 优先扣冷却中 GAME。
- 扣减对应 locked 记录。
- 建议 FIFO，优先扣最早解锁记录。
- 被消费部分标记 consumed。
- 不产生贡献值。
- 不产生即时解锁额度。

## 17. 提现系统

### 17.1 模式

采用：

```text
小额自动 + 大额人工审核
```

### 17.2 限额

前期建议：

| 限制项 | 数值 |
|---|---:|
| 单笔自动提现上限 | 5,000 GAME |
| 单用户每日自动提现上限 | 20,000 GAME |
| 全服每小时自动提现上限 | 30,000 GAME |
| 全服每日自动提现上限 | 150,000 GAME |
| 超过单笔上限 | 人工审核 |
| 超过小时/日额度 | 排队或人工审核 |

按初始价格 `0.0001U / GAME`：

- 单笔 5,000 GAME = 0.5U。
- 单用户每日 20,000 GAME = 2U。
- 全服每小时 30,000 GAME = 3U。
- 全服每日 150,000 GAME = 15U。

### 17.3 审核触发

以下进入人工审核：

- 大额提现。
- 新号提现。
- 多账号关联。
- 同 IP / 设备集群。
- 高频提现。
- 短时间异常产出。
- 交易行异常对倒。
- 风控分过低。

## 18. 外部 GAME 充值

玩家可从市场买 GAME 后充值进游戏。

规则：

- 不进入冷却。
- 可立即消费。
- 系统消费计入贡献等级。
- 系统消费的 10% 形成即时解锁额度。
- 交易行买卖不计贡献。
- 玩家间转账不计贡献。

这是鼓励买盘进入游戏消耗的核心路径：

```text
市场买 GAME -> 充值进游戏 -> 系统消费 -> 提升贡献等级 -> GAME 被 burn/回收/奖励
```

## 19. 测试玩家奖励

采用：

```text
绑定权益为主，少量 GAME 分期给
```

普通测试奖励：

- 绑定道具。
- 宝石余额。
- 白名单。
- 称号。
- NFT Mint 折扣。
- 少量 GAME。

高质量贡献：

- 高质量反馈。
- 内容创作。
- 核心拉新。
- 社区贡献。

高质量贡献奖励从 Growth Wallet 人工发放。

规则：

- 不做大额无条件空投。
- GAME 进入正常冷却。
- 奖励分 2-3 期。
- 按留存和反馈质量解锁。
- 人工奖励必须记录原因和证据。

## 20. KOL / 推手

采用：

```text
小额现金 + 分期 GAME + 结果奖励
```

规则：

- 小 KOL：少量 U 或宝石礼包 + 邀请码。
- 中 KOL：部分 U + locked GAME，按效果分期。
- 大 KOL：前期慎用。
- 推手按有效行为结算。
- GAME 从 Growth Wallet 发。
- 不给一次性大量 unlocked GAME。

有效行为：

- 有效注册。
- 2-3 日活跃。
- 7 日留存。
- 有效充值。
- 外部 GAME 充值。
- 系统消费。
- 交易行成交。

## 21. 邀请系统

按有效行为分层奖励

上线自动邀请前必须具备：

- 反刷规则。

## 22. 风控

采用轻风控：

```text
设备/IP/行为识别 + 提现限额 + 人工审核
```

不做强 KYC。

### 22.1 风控维度

- 设备。
- IP。
- 钱包。
- 登录时间。
- 在线时长。
- 操作路径。
- 产出效率。
- 提现频率。
- 交易行行为。
- 邀请关系。

### 22.2 新号限制

新号前 3-7 天：

- 提现额度更低。
- 高级掉落受限。
- 高级副本受限。
- 高价值交易受限。
- 更容易进入人工审核。

### 22.3 多账号处理

同设备 / 同 IP / 同钱包簇：

- 共享部分提现风控额度。
- 高收益行为降权。
- 异常集中提现人工审核。
- 严重作弊冻结收益。

## 27. 风险边界

必须避免：

- 承诺固定收益。
- 承诺保本。
- 承诺币价上涨。
- 无限制开放提现。
- 热钱包放大额资金。
- 内部人持有大量 unlocked GAME。
- Growth 或 Treasury 无记录转出。
- 用项目方资金频繁主观控盘。
- 无风控邀请奖励。

对外表达应强调：

- GAME 是游戏生态 Token。
- 经济参数会根据测试数据调整。
- Revenue 回流是运营策略，不是收益承诺。
- 游戏奖励受上限、冷却和风控限制。
- 项目会尽量公开关键钱包和基础经济数据。

## 29. 一句话闭环

项目经济闭环如下：

```text
玩家玩游戏 -> 获得 Gold/材料/稀有物
Gold 受限兑换 GAME
GAME 进入冷却
GAME 可用于成长、战斗、交易行、NFT Mint
GAME 消费产生 burn、回收和奖励
宝石充值和 NFT 版税进入 Revenue
Revenue 支持运营、回购、LP、Treasury、活动
提现通过冷却、限额和风控控制节奏
```

V0.2 的核心不是让所有人一进来就赚很多，而是先做到：

- 玩家有东西玩。
- 玩家有资产感。
- 项目有收入。
- Token 有消耗。
- LP 不被快速抽干。
- 参数能根据数据调整。
